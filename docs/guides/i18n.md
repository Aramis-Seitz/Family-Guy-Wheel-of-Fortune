# Internationalisierung (i18n)

Die Oberfläche unterstützt Deutsch (`de`) und Englisch (`en`) mit i18next. Englisch ist die Rückfallsprache. Die Auswahl wird im Browser gespeichert und zusätzlich als `Accept-Language` an die API gesendet.

## Übersetzungen hinzufügen

Neue Schlüssel müssen in beiden Dateien ergänzt werden:

- `public/locales/en/common.json`
- `public/locales/de/common.json`

Schlüssel werden thematisch gruppiert, beispielsweise `shop.purchaseSuccess`. Anschließend prüft `npm run check:i18n` gültiges JSON, Schlüsselparität, leere Übersetzungen und übereinstimmende Interpolationsvariablen.

## Statische HTML-Inhalte

- `data-i18n="key"`: Textinhalt
- `data-i18n-placeholder="key"`: Eingabe-Platzhalter
- `data-i18n-title="key"`: Tooltip/Titel
- `data-i18n-html="key"`: vertrauenswürdiges HTML aus den lokalen Übersetzungsdateien

`data-i18n-html` darf nicht für Benutzereingaben oder externe Inhalte verwendet werden.

## Dynamische Inhalte

In TypeScript wird `t()` aus `public/script/app/i18n.ts` verwendet. Werte werden über i18next interpoliert, zum Beispiel `t("wheel.saved", { name })`. Pluralformen verwenden `count` und passende Schlüssel wie `_one` und `_other`.

Module mit bereits gerenderten dynamischen Inhalten können auf `app:language-changed` reagieren. Statische `data-i18n`-Elemente werden vom zentralen Sprachumschalter automatisch aktualisiert.

## Zahlen, Datum und Uhrzeit

Für sichtbare Werte werden ausschließlich die Hilfsfunktionen aus `public/script/app/format.ts` verwendet:

- `formatNumber()`
- `formatDate()`
- `formatTime()`

Sie nutzen die aktuell aufgelöste Sprache. Fest codierte Locales wie `de-DE` und manuell vorformatierte Zeitstrings sind zu vermeiden.

## API-Fehler

Die API behält das rückwärtskompatible Feld `error` und liefert zusätzlich einen stabilen `code` sowie optional `details`. Der Client übersetzt bekannte Codes über `public/script/api/api-helpers.ts`. Neue Fehlercodes müssen in `server/src/lib/error-codes.ts`, in beiden Locale-Dateien und in der Client-Zuordnung ergänzt werden.

Die API wertet `Accept-Language` aus und setzt `Content-Language`. Benutzertexte werden weiterhin im Frontend übersetzt; Servermeldungen dienen nur als kompatibler Fallback.

## Review-Checkliste

1. Beide Locale-Dateien aktualisieren.
2. Keine Benutzertexte hart im HTML oder TypeScript ergänzen.
3. Interpolation und Pluralisierung prüfen.
4. Locale-abhängige Werte mit den Formatierungshelfern rendern.
5. `npm run check:i18n`, `npm run typecheck` und `npm run build` ausführen.
6. Sprachwechsel auf der betroffenen Seite in beiden Richtungen prüfen.
